# Client Queue Design

## Problem

Today `Client[I,O]` is a thin pass-through: `Stream`/`Invoke` go straight to the
session. The TUI must manage its own "is a turn running?" flag and silently drops
any message submitted mid-turn. This means:

- Messages typed while the agent is working are lost.
- Every client (TUI, future CLI, future cloud runner) has to re-implement the same
  guard logic.
- There is no way for a user to view, reorder, or cancel queued messages.
- Interrupt semantics are ambiguous — does it also clear pending messages?

The fix is to promote the queue into `Client` so all consumers get correct
behaviour for free.

---

## Design

### Turn identity

Each submitted input is wrapped in a `QueuedItem[I]` with a stable ID:

```go
type QueuedItem[I any] struct {
    ID    string // uuid, stable across Pending() calls
    Input I
}
```

Events emitted by `Client` are wrapped to carry which item they belong to:

```go
type TurnEvent[O any] struct {
    ItemID string
    Event  O
}
```

The TUI uses `ItemID` to match events to history entries — a blinking square on
the active item, a solid square on completed items.

### New Client API

`Stream` and `Invoke` are removed from the public surface. The push-based
submit/events pair becomes the sole API:

```go
// Submit enqueues input. If no turn is running, the item is dispatched immediately.
// Returns the stable ID assigned to this item.
func (c *Client[I, O]) Submit(input I) string

// Pending returns a snapshot of all queued (not-yet-started) items in order.
func (c *Client[I, O]) Pending() []QueuedItem[I]

// CancelQueued removes the item with the given ID from the pending queue.
// Returns false if the item is not found (already started or never existed).
func (c *Client[I, O]) CancelQueued(id string) (QueuedItem[I], bool)

// Events returns the channel of turn events. Each value carries the ItemID of
// the originating Submit call and the underlying event.
func (c *Client[I, O]) Events() <-chan TurnEvent[O]

// Interrupt stops the current turn. The next queued item starts immediately.
// Does NOT clear the pending queue.
func (c *Client[I, O]) Interrupt()

// Shutdown stops the current turn and clears the pending queue, then tears down
// the underlying session. Blocks until the session exits.
func (c *Client[I, O]) Shutdown()
```

### Internal mechanics

`Client` owns:
- `pending []QueuedItem[I]` — protected by a mutex
- `events chan TurnEvent[O]` — buffered, cap 256
- a single "dispatcher" goroutine started in `New`

The dispatcher loop:
1. Blocks until a `QueuedItem` is available (signalled via a `chan struct{}`).
2. Calls `sess.Stream(ctx, item.Input)`.
3. Forwards each event as `TurnEvent{ItemID: item.ID, Event: ev}` onto `c.events`.
4. On terminal event (`TurnDone`, `TurnFailed`, `TurnInterrupted`, `TurnRejected`),
   pops the next pending item and repeats.

`Submit` appends to `pending` under the mutex and sends to the signal channel
(non-blocking — the dispatcher may already be running).

`Interrupt` calls `sess.Interrupt()`. The dispatcher sees `TurnInterrupted`,
emits it, and immediately starts the next pending item.

`Shutdown` calls `sess.Shutdown()` and closes `c.events` after the dispatcher exits.

### TUI changes

`Screen` drops:
- `running bool` — replaced by watching for `TurnStarted`/terminal events on
  `TurnEvent.ItemID`
- `pending []string` (from the reverted change)
- the `drain` goroutine — replaced by a single goroutine reading `client.Events()`

`Screen` gains:
- per-message status map: `map[string]itemStatus` keyed by `QueuedItem.ID`
- up-arrow binding: calls `client.CancelQueued(id)` for the top pending item and
  puts the text back in the textarea

### Interrupt UX

`esc` calls `client.Interrupt()`. The current turn stops; the next queued message
starts automatically. The interrupted turn is rendered with a tombstone
(`RoleInterrupted`). Pending items are untouched.

To clear the entire queue the user presses `esc` repeatedly — one interrupt per
queued item — or a future `/clear-queue` slash command.

---

## Out of scope

- Priority reordering of the queue (move item N to top) — add when there is a
  concrete use case.
- Per-item timeout — add when tools are in and long-running turns are observable.
- Persisting the queue across restarts — session state is ephemeral for now.
