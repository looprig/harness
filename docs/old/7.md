# Tools Design

## Scope

This spec covers the 12 built-in tools for the Nexus agent CLI, the permission model that
governs them, workspace containment, and error handling. Subagent wiring (`SubagentTool`
spawning a child `AgentSession`) is covered in `2026-06-05-subagents-design.md`.

Concrete tool implementations live in the root `tools/` package. Each implements `tool.InvokableTool`
from `internal/tool/`. `agents/coding/agent.go` is the agent manifest: it imports from `tools/` and
declares which tools the coding agent gets, alongside model and system prompt.

---

## Architecture context

The current wiring, for reference when reading this spec:

```
cmd/cli/main.go
  → GET /v1/catalog → []catalogAgent{Name, Description}
  → client.NewRegistry()
  → cwd, _ := os.Getwd()   // workspace root
  → for each agent: client.Register(reg, name, gatewayURL, tokenFn, loop.ToolDeps{WorkspaceRoot: cwd, ...})
       → internally: coding.New(gatewayURL, tokenFn, deps) → agent.Agent
       → reg.r.Register(name, agent.Agent)
  → client.New[*client.Input, client.Event](ctx, reg, name)
       → session.NewAgent(ctx, agent.Agent)
            → loop.New(ctx, sessionID, loop.Config)   // starts listen goroutine
       → returns *client.Client[*client.Input, client.Event]
  → tui.New(client, &prog)
       → on user input: client.Stream(ctx, client.NewInput(text))
            → agentSessionAdapter.Stream → sends loop.StartTurn{Input: blocks}
            → returns StreamReader[loop.Event]
       → drain goroutine: reads events, sends to tea.Program via (*prog).Send(eventMsg{ev})
```

`loop.Loop` exposes two channels:
- `Commands chan<- Command` — sealed interface; concrete types: `StartTurn`, `Shutdown`, `Interrupt`, `ApproveToolCall`, `DenyToolCall`, `ProvideUserInput`
- `Events <-chan Event` — sealed interface; concrete types: `SessionStarted`, `TurnStarted`,
  `TurnDone`, `TurnFailed`, `TurnRejected`, `TurnInterrupted`, `TokenDelta`,
  `PermissionRequested`, `UserInputRequested`, `ShellSessionStarted`, `ShellSessionEnded`,
  `ShellSessionOutputDelta`

`loop.Config` is the primary configuration surface. After tools are added it gains one field:

```go
type Config struct {
    Model      llm.ModelSpec
    GatewayURL string
    Token      func() (string, error) // called before each request; owns refresh logic
    HTTPClient *http.Client           // nil = 30s default
    Tools      ToolDeps               // zero value = no tools registered
}
```

Tools integration adds tool dispatch to the turn runner inside `internal/agent/loop/turn.go`.
When the LLM response contains tool calls, `runTurn` invokes the runner, collects results,
appends them to `messages`, and makes another LLM call — the loop continues until the model
responds with no tool calls. This is transparent to the caller; the caller still sees
`TokenDelta`, `TurnDone`, etc.

## Error handling contract

**All tool errors are returned to the model as tool result strings, never as agent-terminating
errors.** The agent loop continues regardless of what a tool returns.

This covers:
- Invalid arguments (JSON unmarshal failure, missing required fields) → `"error: invalid arguments: <detail>"`
- Permission denied → `"error: permission denied: <reason>"`
- Tool execution error (file not found, command failed, network error) → `"error: <detail>"`
- Panic inside tool → recovered via `defer/recover`, returned as `"error: tool panicked: <detail>"`

The only conditions that terminate a turn are: context cancellation (`ctx.Done()`) and
explicit `Interrupt` command. Everything else produces an error result the model can
reason about and retry.

This contract is enforced in a tool runner called from `runTurn` in `internal/agent/loop/turn.go`,
not inside individual tools. Individual tools return `(*tool.ToolResult, error)` — the runner
converts `error` to a text result.

**Permissions do not apply to graph-based workflows** (`internal/graph.ToolsNode`). The
`ToolsNode` is used in operator-controlled deterministic workflows, not open-ended model-driven
loops. `tools/` and `internal/graph/tool_node.go` are separate dispatch layers sharing only
the `tool.InvokableTool` interface.

## Extensibility rules

These rules exist because the coding-agent landscape moves fast. They are the guard rails
that let new features land without touching existing tools.

### Rule 1 — capabilities via optional interfaces, never via BaseTool fields

`BaseTool`, `InvokableTool`, and `StreamableTool` are:

```go
// ToolInfo describes a tool for model registration.
type ToolInfo struct {
    Name       string
    Desc       string
    Parameters map[string]any // JSON Schema-compatible
}

// BaseTool is the minimum interface every tool must satisfy.
// It never gains new methods — new behaviour is added via optional interfaces.
type BaseTool interface {
    Info(ctx context.Context) (*ToolInfo, error)
}

// InvokableTool executes synchronously and returns a ToolResult.
// Used by both the graph ToolsNode and the agent loop turn runner.
type InvokableTool interface {
    BaseTool
    InvokableRun(ctx context.Context, argsJSON string) (*ToolResult, error)
}

// StreamableTool executes and streams string chunks.
// Currently used by the graph ToolsNode only.
type StreamableTool interface {
    BaseTool
    StreamableRun(ctx context.Context, argsJSON string) (*stream.StreamReader[string], error)
}
```

`BaseTool` never gains new methods. New behaviour is added by defining a new optional interface
and having the runner check it via type assertion:

```go
// example: future cache hint
type Cacheable interface {
    CacheKey(argsJSON string) string  // empty = not cacheable for these args
}
```

Runner: `if c, ok := t.(Cacheable); ok { key = c.CacheKey(args) }`.

**The wrong way** is adding `Cacheable() bool` to `BaseTool`. That requires every existing
tool to implement the method. With a new optional interface, existing tools are untouched.

`Sequential`, `EffectChecker`, and `PermissionPrompter` (all in this spec) are the three
live examples of this pattern. Every future capability follows the same shape.

The turn runner in `loop/turn.go` dispatches via `InvokableTool.InvokableRun`. The graph
`ToolsNode` dispatches via the same interface for invokable tools, and via
`StreamableTool.StreamableRun` for streaming tools. All 12 built-in agent tools implement
`InvokableTool`.

### Rule 2 — inject dependencies via ToolDeps, never via individual constructors

Every tool is constructed with a single `ToolDeps` argument. `ToolDeps` lives in
`internal/agent/loop/` (alongside `Config`) so that `loop.Config` can embed it without
creating an import cycle — `tools/` imports `loop/`, not the other way round.

```go
// ToolDeps holds everything a tool might need at construction time.
// Tools accept this struct and ignore fields they don't use.
// Adding a new shared dependency = adding one field here;
// no tool constructor signatures change.
//
// Defined in internal/agent/loop/ alongside Config so that loop.Config can
// embed it directly without creating an import cycle with tools/.
type ToolDeps struct {
    WorkspaceRoot string
    SessionCtx    context.Context     // cancelled when the agent session shuts down
    Permission    PermissionGate      // interface; satisfied by *tools.PermissionChecker
    Events        chan<- Event         // loop.Event channel; tool-emitted events go here
    Registry      []tool.InvokableTool // all registered tools; runner looks up by Info().Name
    Middlewares   []tool.ToolMiddleware
}

// PermissionGate is the narrow interface loop/ needs from the permission system.
// Defined here (consumer side) so loop/ does not import tools/.
// Satisfied by *tools.PermissionChecker.
type PermissionGate interface {
    Check(ctx context.Context, t tool.InvokableTool, toolName, argsJSON string) Effect
    // AddSessionPolicy appends a ToolPolicy for the remainder of the session.
    AddSessionPolicy(p ToolPolicy)
}
```

`cmd/cli/main.go` sets only `WorkspaceRoot` on the `loop.ToolDeps` shell and passes it to
`coding.New`. `coding.New` (the agent manifest) constructs both `tools.NewPermissionChecker`
and the full `[]tool.InvokableTool` registry before sealing `loop.Config`. `cmd/cli/main.go`
does not know about `tools/` at all.

Two fields are wired internally by `loop.New` before starting the listen goroutine: `Events`
is set to the write-side of the events channel, and `SessionCtx` is set to the `ctx` passed
to `loop.New`. Tools that spawn background goroutines (e.g. `ShellSessionTool`) use
`SessionCtx` as their shutdown signal, independent of per-turn contexts which are cancelled
at turn end.

`WorkspaceRoot` is derived from `os.Getwd()` at startup in `cmd/cli/main.go` — no user
prompt needed. The user `cd`s into their project before running `nexus`, so cwd is the
workspace by convention.

Example: when a rate limiter is added, add `Limiter *rate.Limiter` to `ToolDeps`. `Bash`
reads it; the other 11 tools ignore it. No constructor signatures change anywhere.

### Rule 3 — cross-cutting concerns via middleware, never in the runner body

The runner exposes a `Middleware` chain for behaviour that wraps every tool call:

```go
// ToolMiddleware wraps a tool call. next is the actual InvokableRun call.
// Middleware is applied in the order listed in ToolDeps.Middlewares;
// the first middleware is outermost.
type ToolMiddleware func(ctx context.Context, tool InvokableTool, argsJSON string, next ToolExecuteFunc) (*ToolResult, error)

type ToolExecuteFunc func(ctx context.Context, argsJSON string) (*ToolResult, error)
```

`ToolDeps.Middlewares []ToolMiddleware` — supplied at construction. Examples of
behaviour that belongs here, not in the runner body:

| Concern | Middleware | Not in runner |
|---|---|---|
| OpenTelemetry spans | wrap with `tracer.Start` / `span.End` | ✓ |
| Per-tool rate limiting | check `rate.Limiter` before `next` | ✓ |
| Audit logging | record call+result to journal | ✓ |
| Result caching | check cache before `next`, store after | ✓ |
| Timeout per tool type | wrap `next` with `context.WithTimeout` | ✓ |

The runner body itself stays thin: partition sequential/parallel, acquire semaphore, call
the middleware chain, collect results. Nothing else.

### ToolResult — flexible return type

Tools return `*tool.ToolResult` instead of a plain string. This allows tools to return images,
structured content, or signal turn termination — without changing the `BaseTool` interface
for simple text-only tools.

```go
// ToolResult is what every tool InvokableRun returns.
// Content holds content.Block values, which the runner wraps into a
// FunctionToolResult block before appending to the conversation history.
type ToolResult struct {
    // Content is what the model sees in the tool result message.
    // Typically one TextBlock, but can include ImageBlock.
    // Must have at least one block. Runner injects "error: empty result" if nil.
    Content []*content.Block

    // Terminate signals the agent loop to end the turn after this tool batch
    // completes, without making another LLM call.
    Terminate bool
}

// TextResult is a convenience constructor for the common case: one text block,
// Terminate false.
func TextResult(s string) *ToolResult {
    return &ToolResult{Content: []*content.Block{{Type: content.TypeText, Text: s}}}
}
```

`content.Block` lives in `internal/content/` — a pure data package imported by both
`internal/tool/` and `internal/llm/`. This keeps the dependency graph clean: `internal/tool/`
does not depend on `internal/llm/`.

```
internal/content/    ← pure data (Block, BlockType, ImageBlock, etc.)
    ↑           ↑
internal/tool/   internal/llm/
    ↑
internal/agent/loop/   (ToolDeps, PermissionGate, Effect, ToolPolicy)
    ↑
tools/                 (implements those contracts; defines the 12 tools)
    ↑
agents/coding/         (agent manifest: imports tools/, declares tool set)
```

**Thinking blocks in tool-use turns:** `content.Block` has a `TypeThinking` variant. When
assembling an LLM response that contains tool calls, the runner must preserve any thinking
blocks in the assistant message before the tool results — they are part of the message and
required for extended thinking models. Do not strip them. Pass them through as-is when
appending to `messages`.

**Terminate semantics in the turn loop:** after the parallel batch completes, if any result
has `Terminate: true`, `runTurn` does not make another LLM call. It emits `TurnDone` with
the assembled content from the last assistant message. Tool results are still appended to
history so the full exchange is preserved.

### Sequential — serialised execution marker

Some tools must not run concurrently with other tool calls in the same LLM response. They
implement the `Sequential` interface:

```go
// Sequential is optionally implemented by a BaseTool to signal that it must not
// run in parallel with other tool calls. The tool runner places sequential tools
// in a serial batch that completes before any parallel calls begin.
type Sequential interface {
    Sequential() bool
}
```

The runner checks via interface assertion — no allowlists or name matching in the runner:

```go
func isSequential(t tool.BaseTool) bool {
    s, ok := t.(tool.Sequential)
    return ok && s.Sequential()
}
```

`ShellSessionTool` implements `Sequential() bool { return true }` because shell sends are
stateful and must not interleave. All other built-in tools do not implement this interface
and run in the parallel batch.

### EffectChecker — per-call effect override

Some tools need to return a different `Effect` depending on their arguments, not just their
name. `PermissionChecker.Check` tests this interface as Stage 0 before any policy lookup:

```go
// EffectChecker is optionally implemented by tools that need to override the permission
// effect based on their own arguments.
// Lives in tools/permission.go; uses loop.Effect from loop/deps.go.
type EffectChecker interface {
    CheckEffect(argsJSON string) (effect loop.Effect, handled bool)
}
```

The runner calls `Permission.Check(ctx, tool, toolName, argsJSON)` once. Inside `Check`, Stage 0 runs first:
```
1. If tool implements EffectChecker: call CheckEffect(argsJSON)
   → if handled: return that effect immediately; remaining stages are skipped
2. ...remaining stages (containment, hard rules, persisted approvals, session policies)
```

`ShellSessionTool` is the only built-in tool that implements `EffectChecker`. See the
ShellSession section below for details.

`Effect` and `ToolPolicy` live in `internal/agent/loop/deps.go` (alongside `ToolDeps`) so
`loop/` can reference them without importing `tools/`. `EffectChecker` lives in
`tools/permission.go` as it is a tool-side fast-path interface. If a second consumer outside
of `tools/` needs `EffectChecker`, move it to `internal/tool/` at that point. Do not move it
speculatively.

## Permission model

Tool execution passes through a seven-stage evaluation. Each stage is represented by a
distinct type. Stages run top-to-bottom; once a stage returns a definitive effect the
remaining stages are skipped.

```
Stage 0  EffectChecker      — optional per-call override from the tool itself
Stage 1  ContainmentCheck   — workspace boundary; always runs first for file tools
Stage 2  HardApproveRules   — operator-set always-allow; user cannot override
Stage 3  HardDenyRules      — operator-set always-deny; user cannot override
Stage 4  PersistedApprovals — user-saved rules (workspace-level, then user-level)
Stage 5  SessionPolicies    — per-session rules from PermissionPolicy.Policies
Stage 6  DefaultEffect      — EffectAsk if nothing matched
```

### Effect

```go
type Effect uint8

const (
    EffectAutoApprove Effect = iota // execute without prompting
    EffectAsk                       // emit PermissionRequested event, block until user responds
    EffectDeny                      // reject immediately; error result returned to model
)
```

### Stage 1 — ContainmentCheck

Not a configurable struct — always enforced before any policy lookup. Defined by
`containedPath`:

```go
// containedPath resolves input relative to workspaceRoot, cleans it, and
// verifies it remains within the workspace.
func containedPath(workspaceRoot, input string) (string, error)
```

Applies to: `ReadFile`, `WriteFile`, `EditFile`, `Glob`, `Grep`, `Bash` (workdir),
`ShellSession` (workdir). Non-file tools skip this stage.

### Stage 2 — HardApproveRules

```go
type HardApproveRules struct {
    // Tools is a list of exact tool names that are always auto-approved.
    // "*" matches all tools.
    Tools []string
}
```

### Stage 3 — HardDenyRules

```go
type HardDenyRules struct {
    // DeniedReadaths are absolute glob patterns that can never be read.
    // Defaults: ~/.ssh/**, ~/.gnupg/**, **/.env, **/.env.*, **/id_rsa,
    //           **/id_ed25519, **/*.pem, **/*.key, **/credentials, **/secrets
    DeniedReadaths []string

    // DeniedWritePaths are absolute glob patterns that can never be written.
    // Defaults: same as DeniedReadaths plus **/.git/config, **/go.sum
    DeniedWritePaths []string

    // DeniedBashPrefixes are command prefixes that can never run via Bash or ShellSession.
    // Defaults: "rm -rf /", "sudo", "su ", "curl | bash", "wget | bash",
    //           "chmod 777", "> /dev/sd", "dd if=", "mkfs"
    DeniedBashPrefixes []string

    // MaxReadBytes caps the bytes ReadFile may return per call. Default: 1MB.
    MaxReadBytes int64
}
```

### Stage 4 — PersistedApprovals

User-saved approval rules read from disk on every permission check. Re-reading on each
check (rather than caching at session start) is intentional: when the user presses `w` to
approve a tool at workspace scope during one permission gate, the next tool call's Check
sees that approval on disk immediately without needing to restart the session.

The file bytes are read on every check, but parsing is skipped when content is unchanged.
`PermissionChecker` holds two cache instances — one per file — each wrapping `sha256`-keyed
lazy parsing so JSON unmarshalling is skipped when the file bytes haven't changed. This
helper is `internal/hashcache/` — already in the repo. It has one type: `Cache[T]` with a
`Load(content []byte) (T, error)` method (parse func is supplied at construction via `hashcache.New`).

```go
type ApprovalRecord struct {
    Tool          string `json:"tool"`
    PathPattern   string `json:"path_pattern,omitempty"`
    CommandPrefix string `json:"command_prefix,omitempty"`
    Effect        Effect `json:"effect"`
}

type ApprovalsFile struct {
    Version   int              `json:"version"`   // current: 1
    Approvals []ApprovalRecord `json:"approvals"` // first match wins
}
```

**File locations:**

| Priority | Path | Written when |
|---|---|---|
| 1 (higher) | `<workspace>/.nexus/approvals.json` | user presses `w` at approval prompt |
| 2 (lower)  | `~/.nexus/approvals.json`           | user edits the file directly |

### Stage 5 — SessionPolicies

```go
type ToolPolicy struct {
    Tool     string   // exact tool name or "*"
    Effect   Effect
    Paths    []string // workspace-relative glob patterns; empty = all paths
    Commands []string // allowed command prefixes; empty = all commands
}
```

Populated from `PermissionPolicy.Policies` at session construction. Extended at runtime when
the user chooses scope `s` (session) at the approval prompt.

### PermissionPolicy — the complete config struct

```go
type PermissionPolicy struct {
    WorkspaceRoot string
    HardApprove   HardApproveRules
    HardDeny      HardDenyRules
    Policies      []ToolPolicy // extended in-place at runtime for session-scope approvals
}
```

`PermissionPolicy` lives in `ToolDeps.Permission` as a `*PermissionChecker` (see below).
It is constructed by the CLI or gateway before the agent is started and stored in `loop.Config`
via `ToolDeps`.

### PermissionChecker — the stateful runtime object

```go
type PermissionChecker struct {
    mu        sync.RWMutex
    policy    PermissionPolicy
    wsCache   *hashcache.Cache[ApprovalsFile] // <workspace>/.nexus/approvals.json
    userCache *hashcache.Cache[ApprovalsFile] // ~/.nexus/approvals.json
}

func NewPermissionChecker(policy PermissionPolicy) *PermissionChecker
```

`PermissionChecker` satisfies the `loop.PermissionGate` interface. `Check` runs the
seven-stage evaluation. `AddSessionPolicy` appends to `policy.Policies` under `mu`.

### Policy evaluation order

```
(c *PermissionChecker) Check(ctx, tool InvokableTool, toolName string, argsJSON string) Effect:

  // Stage 0: optional per-call effect override (EffectChecker interface)
  if tool implements EffectChecker:
    effect, handled := tool.CheckEffect(argsJSON)
    if handled: return effect

  // Stage 1: workspace containment
  // File tools (ReadFile, WriteFile, EditFile, Glob, Grep): check path arg.
  // Bash, ShellSession: check workdir arg when non-empty.
  if toolName has a path or workdir argument:
    _, err := containedPath(c.policy.WorkspaceRoot, extractPathOrWorkdir(argsJSON))
    if err != nil: return EffectDeny

  // Stage 2: hard approve
  for name in c.policy.HardApprove.Tools:
    if name == "*" || name == toolName: return EffectAutoApprove

  // Stage 3: hard deny
  if file tool && path matches DeniedReadaths/DeniedWritePaths: return EffectDeny
  if bash tool && command matches DeniedBashPrefixes: return EffectDeny

  // Stage 4: persisted approvals
  wsFile,   _ := c.wsCache.Load(os.ReadFile("<workspace>/.nexus/approvals.json"))
  userFile, _ := c.userCache.Load(os.ReadFile("~/.nexus/approvals.json"))
  for record in append(wsFile.Approvals, userFile.Approvals...):
    if record matches: return record.Effect

  // Stage 5: session policies
  for p in c.policy.Policies:
    if p matches: return p.Effect

  // Stage 6: default
  return EffectAsk
```

### PermissionPrompter — typed permission requests

Tools that default to `EffectAsk` implement `PermissionPrompter` to declare exactly what
kind of question to ask the user. The runner checks for this interface before emitting
`PermissionRequested`:

```go
// PermissionPrompter is optionally implemented by tools that need user approval.
// BuildRequest is called only when PermissionChecker.Check returns EffectAsk —
// i.e. after the permission check, not before. It receives the same argsJSON
// that Check evaluated and returns the typed request the TUI will render.
// Lives in internal/tool/ alongside BaseTool so loop/ can reference it without
// importing tools/.
type PermissionPrompter interface {
    BuildRequest(argsJSON string) (PermissionRequest, error)
}

// PermissionRequest is the sealed interface for all typed permission prompts.
// The TUI calls ToolName() for the header, Description() for the question body,
// and AllowedScopes() for the approval key options — no type switch needed for rendering.
// Deny (n) is always available and is never part of AllowedScopes — it maps to
// DenyToolCall, not ApproveToolCall, and is handled separately by the runner.
type PermissionRequest interface {
    permissionRequest()
    // ToolName returns the name of the tool requesting permission,
    // shown as the prompt header. e.g. "WriteFile", "Bash", "WebSearch".
    ToolName() string
    // Description returns the human-readable question body shown below the header.
    // e.g. "Write src/main.go?" or "Run: go test ./...?"
    Description() string
    // AllowedScopes returns the approval scopes offered as key options (y/s/w).
    // Deny (n) is always appended by the renderer and is not in this list.
    AllowedScopes() []ApprovalScope
}

// ApprovalScope is the persistence duration the user chooses when approving a tool call.
// Lives in internal/tool/ alongside PermissionRequest so loop/ and client/ can use it
// without importing tools/.
type ApprovalScope uint8

const (
    ScopeOnce      ApprovalScope = iota // approve this call only; nothing persisted
    ScopeSession                        // approve for remainder of session; in-memory policy
    ScopeWorkspace                      // approve for this workspace; written to .nexus/approvals.json
)
```

**Concrete request types** (all live in `tools/permission.go`):

```go
// FileWriteRequest is raised by WriteFile and EditFile.
type FileWriteRequest struct {
    Tool string // "WriteFile" or "EditFile" — set by BuildRequest
    Path string // workspace-relative path being written
}

func (FileWriteRequest) permissionRequest() {}
func (r FileWriteRequest) ToolName()    string { return r.Tool }
func (r FileWriteRequest) Description() string {
    verb := "Write"
    if r.Tool == "EditFile" {
        verb = "Edit"
    }
    return verb + " " + r.Path + "?"
}
func (FileWriteRequest) AllowedScopes() []ApprovalScope {
    return []ApprovalScope{ScopeOnce, ScopeSession, ScopeWorkspace}
}

// BashRequest is raised by Bash.
type BashRequest struct {
    Command string // full command string
    Workdir string // workspace-relative workdir, empty = workspace root
}

func (BashRequest) permissionRequest() {}
func (BashRequest) ToolName()           string { return "Bash" }
func (r BashRequest) Description()      string { return "Run: " + r.Command + "?" }
func (BashRequest) AllowedScopes() []ApprovalScope {
    return []ApprovalScope{ScopeOnce, ScopeSession, ScopeWorkspace}
}

// ShellSessionStartRequest is raised by ShellSession on action="start".
// Once approved, subsequent sends on the same session are auto-approved via
// EffectChecker — no further prompts.
type ShellSessionStartRequest struct {
    Workdir string // workspace-relative workdir, empty = workspace root
}

func (ShellSessionStartRequest) permissionRequest() {}
func (ShellSessionStartRequest) ToolName() string { return "ShellSession" }
func (r ShellSessionStartRequest) Description() string {
    if r.Workdir == "" {
        return "Start interactive shell?"
    }
    return "Start interactive shell in " + r.Workdir + "?"
}
func (ShellSessionStartRequest) AllowedScopes() []ApprovalScope {
    return []ApprovalScope{ScopeOnce, ScopeSession, ScopeWorkspace}
}

// FetchRequest is raised by Fetch.
type FetchRequest struct {
    Method string // "GET" or "POST"
    URL    string
}

func (FetchRequest) permissionRequest() {}
func (FetchRequest) ToolName()        string { return "Fetch" }
func (r FetchRequest) Description()   string { return r.Method + " " + r.URL + "?" }
func (FetchRequest) AllowedScopes() []ApprovalScope {
    return []ApprovalScope{ScopeOnce, ScopeSession, ScopeWorkspace}
}

// WebSearchRequest is raised by WebSearch.
type WebSearchRequest struct {
    Query string
}

func (WebSearchRequest) permissionRequest() {}
func (WebSearchRequest) ToolName()        string { return "WebSearch" }
func (r WebSearchRequest) Description()   string { return `Search web for "` + r.Query + `"?` }
func (WebSearchRequest) AllowedScopes() []ApprovalScope {
    return []ApprovalScope{ScopeOnce, ScopeSession, ScopeWorkspace}
}
```

Tools that are always `EffectAutoApprove` (ReadFile, Glob, Grep, Todo, AskUser, Subagent)
do not implement `PermissionPrompter`. If a tool does not implement `PermissionPrompter`
but `Check` returns `EffectAsk` (e.g. from a user-configured session policy), the runner
falls back to a generic `UnknownRequest`:

```go
// UnknownRequest is the fallback when a tool implements no PermissionPrompter.
type UnknownRequest struct {
    Tool     string // tool name
    ArgsJSON string // raw args, shown as-is
}

func (UnknownRequest) permissionRequest() {}
func (r UnknownRequest) ToolName()    string { return r.Tool }
func (r UnknownRequest) Description() string { return r.ArgsJSON }
func (UnknownRequest) AllowedScopes() []ApprovalScope {
    return []ApprovalScope{ScopeOnce, ScopeSession, ScopeWorkspace}
}
```

### Permission events and TUI prompt

When `Check` returns `EffectAsk`, the runner builds the typed request, emits
`PermissionRequested` on `ToolDeps.Events`, and blocks on `turn.replies` until a response
arrives.

**callID lifecycle:** `turn.go` generates a `callID` via `randutil.NewUUID()` per tool call
*before* handing it to the runner. It is injected into `ctx` alongside `turn.replies` so that `AskUser`
(which also blocks on `turn.replies`) can share the same channel without ambiguity —
each response is validated against its `callID` before being acted on.

The gate protocol:

1. Runner generates `callID`, injects `callID` and `turn.replies` into `ctx` via
   `context.WithValue` using package-private keys.
2. Runner calls `Permission.Check(ctx, tool, toolName, argsJSON)`.
3. If `EffectAsk`: runner calls `tool.(PermissionPrompter).BuildRequest(argsJSON)` (or
   constructs `UnknownRequest`), then emits `PermissionRequested{CallID, Request}` on
   `ToolDeps.Events`, then blocks on `<-turn.replies` or `<-ctx.Done()`.
4. `loop.listen` receives `ApproveToolCall` / `DenyToolCall` as `controlCommand`s,
   forwards them to `activeTurn.replies` (the `replies chan Command` field on the active
   `Turn` struct, referenced via `loopState.activeTurn`).
5. `runTurn` unblocks, validates `callID` (discards stale responses with a wrong callID),
   resolves the effect.

`Turn` and `loopState` each gain one field to support this:

```go
// Turn gains a replies channel for in-turn control responses (permission gates, AskUser).
type Turn struct {
    cmd       StartTurn
    turnIndex int
    ctx       context.Context
    cancel    context.CancelFunc
    replies   chan Command // buffered(1); runner reads from this; listen writes to this
}

// loopState gains a pointer to the active Turn so handleCommand can reach its replies channel.
type loopState struct {
    turnIndex   int
    status      loopStatus
    cancelTurn  func()
    cancelQueue []func()
    activeTurn  *Turn // non-nil while a turn is running; nil otherwise
}
```

`handleCommand` in `loop.go` is extended to forward the three new control commands:

```go
case ApproveToolCall, DenyToolCall, ProvideUserInput:
    if state.activeTurn != nil {
        select {
        case state.activeTurn.replies <- cmd:
        default: // runner already unblocked or ctx cancelled; drop silently
        }
    }
```

`listen` sets `state.activeTurn` when `turnRunnerStarted` is received and clears it on
`turnRunnerStopped` — matching the existing `cancelTurn` lifecycle.

**`client.PermissionPrompt` — view model (lives in `client/permission.go`)**

`loop.Loop.Commands` is a `chan<- Command` — any command can be sent at any time from
outside, not just during a turn. `Interrupt()` and `Shutdown()` on `AgentSession` already
use this channel. `Approve`, `Deny`, and `ProvideUserInput` are the same pattern: typed
convenience methods that send the appropriate command to the same channel.

`AgentSession` gains three typed methods (consistent with the existing `Interrupt`/`Shutdown`
style — not a generic `SendCommand`):

```go
// Approve sends ApproveToolCall to the loop's Commands channel.
func (s *AgentSession) Approve(callID string, scope tool.ApprovalScope) {
    s.loop.Commands <- loop.ApproveToolCall{CallID: callID, Scope: scope}
}

// Deny sends DenyToolCall to the loop's Commands channel.
func (s *AgentSession) Deny(callID string) {
    s.loop.Commands <- loop.DenyToolCall{CallID: callID}
}

// ProvideUserInput sends ProvideUserInput to the loop's Commands channel.
func (s *AgentSession) ProvideUserInput(callID, answer string) {
    s.loop.Commands <- loop.ProvideUserInput{CallID: callID, Answer: answer}
}
```

`agentSessionAdapter` mirrors these through to the underlying `*session.AgentSession`.
`Session[I,O]` is not extended — workflow sessions have no permission gates.

`Client` gains three matching methods so the TUI never imports loop or session types directly:

```go
// commandSession is the narrow interface agentSessionAdapter satisfies.
// Not part of Session[I,O] — workflow sessions don't need it.
type commandSession interface {
    Approve(callID string, scope tool.ApprovalScope)
    Deny(callID string)
    ProvideUserInput(callID, answer string)
}

func (c *Client[I, O]) Approve(callID string, scope tool.ApprovalScope) {
    if cs, ok := c.sess.(commandSession); ok { cs.Approve(callID, scope) }
}
func (c *Client[I, O]) Deny(callID string) {
    if cs, ok := c.sess.(commandSession); ok { cs.Deny(callID) }
}
func (c *Client[I, O]) ProvideAnswer(callID, answer string) {
    if cs, ok := c.sess.(commandSession); ok { cs.ProvideUserInput(callID, answer) }
}
```

**TUI interaction:** when `eventMsg{ev: loop.PermissionRequested{...}}` arrives:

```
m.prompt = client.BuildPrompt(ev)   // store; replaces input box in View()
```

Input box state is preserved untouched. User presses a **single key** — no enter needed:
- `y` / `s` / `w` → `m.client.Approve(m.prompt.CallID, scope)` → `m.prompt = nil`
- `n` → `m.client.Deny(m.prompt.CallID)` → `m.prompt = nil`
- any other key → ignored

```
┌─────────────────────────────────────────────────┐
│ you: write tests for internal/tool/tool.go      │
│                                                  │
│ assistant: I'll write table-driven tests.       │
│ Let me read the file first…                     │
│                                                  │
├─────────────────────────────────────────────────┤
│ ⠋ running                                       │
├─────────────────────────────────────────────────┤
│ WriteFile                                        │
│ Write internal/tool/tool_test.go?                │
│ [y] once  [s] session  [w] workspace  [n] no    │
└─────────────────────────────────────────────────┘
```

User presses `w` → prompt dismissed, input box restored. If a second tool call in the
same batch also needs permission, its prompt appears immediately after.

On a terminal event (`TurnDone`, `TurnFailed`, `TurnInterrupted`) any active
`m.prompt` is set to nil and the input box is restored — the turn is over.

`ApprovalScope` and its mapping:

| Key | Scope | Behaviour |
|---|---|---|
| `y` | `ScopeOnce` | Unblocks gate; nothing persisted |
| `s` | `ScopeSession` | Appends `ToolPolicy` in-memory; forgotten on exit |
| `w` | `ScopeWorkspace` | Appends `ApprovalRecord` to `<workspace>/.nexus/approvals.json` |
| `n` | Deny | Sends `DenyToolCall`; nothing persisted |

`~/.nexus/approvals.json` (user-level) is not offered at the prompt — users edit it
directly to set cross-project defaults.

## Tool implementations

### ReadFile

```go
type ReadFileArgs struct {
    Path      string `json:"path"`
    StartLine int    `json:"start_line,omitempty"` // 1-based; 0 = from beginning
    EndLine   int    `json:"end_line,omitempty"`   // 1-based inclusive; 0 = to end
}
```

- Resolves path via `containedPath`.
- Checks `HardDeny.DeniedReadaths`.
- Reads file, applying line range if specified.
- Truncates at `HardDeny.MaxReadBytes`; appends truncation notice if cut.
- Returns file contents as string with line numbers prefixed (`"  1\t<line>"`).
- Default policy: `EffectAutoApprove` within workspace.

### WriteFile

```go
type WriteFileArgs struct {
    Path    string `json:"path"`
    Content string `json:"content"`
}
```

- Resolves path via `containedPath`.
- Checks `HardDeny.DeniedWritePaths`.
- Creates parent directories as needed (`os.MkdirAll`).
- Writes atomically: write to `<path>.tmp`, then `os.Rename`.
- Default policy: `EffectAsk`.

### EditFile

str_replace style. The model supplies an exact string to find and its replacement.

```go
type EditFileArgs struct {
    Path       string `json:"path"`
    OldString  string `json:"old_string"`
    NewString  string `json:"new_string"`
    ReplaceAll bool   `json:"replace_all,omitempty"`
}
```

- Resolves path via `containedPath`.
- Checks `HardDeny.DeniedWritePaths`.
- Reads file, counts occurrences of `OldString`:
  - 0 → `"error: old_string not found in <path>"`
  - 2+ and `ReplaceAll` false → `"error: old_string is ambiguous: found N times"`
  - 2+ and `ReplaceAll` true → replace all, write atomically, return count
  - 1 → replace, write atomically
- On success returns a short diff preview (up to 6 lines per side).
- Default policy: `EffectAsk`.

### Bash

```go
type BashArgs struct {
    Command string `json:"command"`
    Workdir string `json:"workdir,omitempty"`
    Timeout int    `json:"timeout_seconds,omitempty"` // default 30, max 120
}
```

- Checks `HardDeny.DeniedBashPrefixes`.
- Resolves `Workdir` via `containedPath` when non-empty.
- Executes via `exec.CommandContext`.
- Captures combined stdout+stderr.
- Non-zero exit code: returned as `"exit status N:\n<output>"`.
- Output truncated at 32KB; appends truncation notice if cut.
- Path warnings (advisory): tokens checked via `filepath.IsAbs`; warns if absolute path is outside workspace.
- Default policy: `EffectAsk`.

### Glob

```go
type GlobArgs struct {
    Pattern string `json:"pattern"`
    Root    string `json:"root,omitempty"`
}
```

- Resolves `Root` via `containedPath`.
- Uses `doublestar.Glob` for `**` support.
- Returns matched paths as newline-separated workspace-relative paths.
- Limits to 500 results; appends notice if truncated.
- Default policy: `EffectAutoApprove`.

### Grep

```go
type GrepArgs struct {
    Pattern      string `json:"pattern"`
    Path         string `json:"path"`
    Recursive    bool   `json:"recursive,omitempty"`
    IgnoreCase   bool   `json:"ignore_case,omitempty"`
    ContextLines int    `json:"context_lines,omitempty"`
    IncludeAll   bool   `json:"include_all,omitempty"`
}
```

- Resolves path via `containedPath`.
- Ripgrep acceleration: if `rg` is present, delegates to it. Falls back to `filepath.WalkDir` + `regexp`.
- Skips by default: `.git/`, `node_modules/`, `vendor/`, `.venv/`, `__pycache__/`, `dist/`, `build/`.
- Returns `"<file>:<line>: <content>"` per match.
- Limits to 200 matches; appends notice if truncated.
- Default policy: `EffectAutoApprove`.

### Fetch

```go
type FetchArgs struct {
    URL     string            `json:"url"`
    Method  string            `json:"method,omitempty"`          // "GET" (default) or "POST"
    Headers map[string]string `json:"headers,omitempty"`
    Body    string            `json:"body,omitempty"`
    Timeout int               `json:"timeout_seconds,omitempty"` // default 15, max 60
}
```

- Method must be `"GET"` or `"POST"`.
- Response body truncated to 64KB.
- Returns `"<status>\n\n<body>"`.
- Default policy: `EffectAsk`.

### WebSearch

```go
type WebSearchArgs struct {
    Query   string `json:"query"`
    Results int    `json:"results,omitempty"` // default 5, max 10
}
```

- Delegates to `SearchProvider` interface (`tools/search.go`).
  First concrete implementation: DuckDuckGo HTML scrape (no API key required).
- Default policy: `EffectAsk`.

### AskUser

Lets the model ask the user a domain question mid-turn. Distinct from `PermissionRequested`
— this is the LLM asking a conversational question, not the system asking for tool approval.

```go
type AskUserArgs struct {
    Question string   `json:"question"`
    Choices  []string `json:"choices,omitempty"`
}
```

- Reads `callID` and `replies chan loop.Command` injected into `ctx` by the turn runner
  before `InvokableRun`. (The runner injects both via `context.WithValue` using package-private
  keys; `AskUser` retrieves them with the same keys.)
- Emits `UserInputRequested{CallID: callID, Question: ..., Choices: ...}` on `ToolDeps.Events`.
- Blocks on `<-replies` or `<-ctx.Done()` until the loop delivers a
  `ProvideUserInput{CallID: callID, Answer: ...}` command; validates `callID` on receipt.
- Answer validation: if `Choices` is non-empty and the answer is not one of the choice
  labels (including "other") → error result returned to model.
- Default policy: `EffectAutoApprove`.

`UserInputRequested` is added to the `loop.Event` sealed interface. `ProvideUserInput` is
added to the `loop.Command` sealed interface as a `controlCommand` — handled immediately by
`listen` and forwarded to `activeTurn.replies`, same as `ApproveToolCall`.

**`client.UserInputPrompt` — view model (lives in `client/ask_user.go`)**

// UserInputPrompt is the view model built from a UserInputRequested event.

    CallID   string
    Question string
    Options  []ChoiceOption // empty = go straight to input box; non-empty = show key list
}

// BuildUserPrompt constructs a UserInputPrompt from a UserInputRequested event.
// Assigns keys a/b/c/… to each choice and appends an "other..." free-text option.
func BuildUserPrompt(ev loop.UserInputRequested) *UserInputPrompt

**TUI interaction:** the LLM question renders as an assistant message in history. The input

*No choices (pure free text):*

│ assistant: Which approach do you prefer —      │
│            refactor or rewrite from scratch?    │
├─────────────────────────────────────────────────┤

├─────────────────────────────────────────────────┤

*With choices:*

### Todo

In-memory only. Zero filesystem access. One tool, three actions.

```go
type TodoArgs struct {
    Action      string `json:"action"`                // "create" | "update" | "list"
    Description string `json:"description,omitempty"`
    ID          string `json:"id,omitempty"`
    Status      string `json:"status,omitempty"`      // "in_progress" | "done" | "cancelled"
}
```

State lives in a `sync.Mutex`-protected map on the `TodoTool` struct itself (one instance
per agent session, constructed once in `NewTodoTool(deps ToolDeps)`). It is not persisted
across sessions.

- `create` → generates UUID, inserts, returns `"created: <id>"`.
- `update` → looks up by ID, updates status, returns `"updated: <id>"`.
- `list` → returns all todos as formatted text grouped by status.
- Default policy: `EffectAutoApprove`.

### ShellSession

Interactive persistent shell session. Two actions: `start` and `send`.

```go
type ShellSessionArgs struct {
    Action    string `json:"action"`
    SessionID string `json:"session_id,omitempty"`
    Command   string `json:"command,omitempty"`
    Workdir   string `json:"workdir,omitempty"`
    Timeout   int    `json:"timeout_ms,omitempty"` // default 5000, max 30000
}
```

- `start`: checks deny rules, resolves workdir, starts shell via `exec.Cmd` with stdin/stdout
  pipes. Background goroutine waits for process exit and emits `ShellSessionEnded` event.
  Stores `shellSessionID` in `approvedSessions`. Returns `"session_id: <id>"`.
- `send`: `CheckEffect` returns `EffectAutoApprove` if session ID is in `approvedSessions`.
  Writes command to stdin pipe, reads output for up to `Timeout` ms. Output truncated at 32KB.
- Implements `Sequential() bool { return true }` — shell sends must not interleave.
- Cleanup: `ShellSessionTool` starts one background goroutine on construction that blocks on
  `<-deps.SessionCtx.Done()` and kills all live processes in the `shells` map. This goroutine
  is owned by the tool and exits when the session context is cancelled (i.e. when `loop.New`'s
  parent context is cancelled or `Shutdown` is sent).

`EffectChecker` implementation:
```go
func (t *ShellSessionTool) CheckEffect(argsJSON string) (loop.Effect, bool) {
    var args ShellSessionArgs
    if json.Unmarshal([]byte(argsJSON), &args) != nil {
        return 0, false
    }
    if args.Action == "send" {
        t.mu.Lock()
        _, approved := t.approvedSessions[args.SessionID]
        t.mu.Unlock()
        if approved {
            return loop.EffectAutoApprove, true
        }
        return loop.EffectAsk, true
    }
    return 0, false // start: fall through to standard policy
}
```

Events emitted: `ShellSessionStarted`, `ShellSessionEnded`, `ShellSessionOutputDelta` —
all added to the `loop.Event` sealed interface in `internal/agent/loop/event.go`.

## How tools wire into the agent loop

`runTurn` in `internal/agent/loop/turn.go` is extended to handle the agentic tool-use loop:

```
runTurn(ctx, cmd StartTurn, turnIndex int, messages, cfg, emit):
  append user message
  emit TurnStarted

  loop:
    stream LLM response via streamViaGateway
      emit TokenDelta for each text chunk as they arrive
    if ctx cancelled: return (no terminal event)
    if response has no tool calls:
      assemble, append to messages, emit TurnDone, return

    // Permission resolution is intentionally sequential across all tool calls
    // in the batch. If the user grants session-scope approval for call N,
    // call N+1's Check sees it immediately and may not need to ask.
    for each tool call in response (sequentially):
      resolve tool by name from cfg.Tools.Registry
      check permission via cfg.Tools.Permission.Check(...)
      if EffectAsk: emit PermissionRequested, block on turn.replies
      if EffectDeny: record tool result = "error: permission denied"
      else: mark approved for execution

    // Execute approved tools (parallel/sequential batching per Sequential interface)
    runner.RunBatch(ctx, approvedCalls, cfg.Tools) → collect results

    append assistant message + tool results to messages
    continue loop
```

`loop.Config.Tools.Registry` is populated in `agents/coding/agent.go` (the agent manifest),
not in `cmd/cli/main.go`. `coding.New(gatewayURL, tokenFn, deps)` calls the tool constructors
from `tools/` and stores the resulting `[]tool.InvokableTool` in `deps.Registry` before
building the sealed `loop.Config`. `cmd/cli/main.go` passes in `deps` (with `WorkspaceRoot`
set from `os.Getwd()`) but does not know which tools exist. The `client.New` path is unchanged
— it still calls `session.NewAgent` which calls `loop.New(ctx, sessionID, cfg)`.
